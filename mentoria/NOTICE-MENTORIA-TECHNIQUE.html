Documentation Technique & Architecture

Spécifications techniques de Mentoria v2.7+

✍️ Conçu et développé par Gilles Cormi, enseignant (académie de Corse) — gcormi.forge.apps.education.fr/frojet/mentoria  ·  v2.7 — Avril 2026

🚧 Work In Progress — Outil en évolution continue

Mentoria est un projet en développement actif (actuellement en version 2.7), conçu selon une approche agile. Les fonctionnalités techniques décrites dans ce document (comme l'utilisation de l'API WebDAV, le proxy n8n, ou la gestion des flux de logs) font l'objet d'optimisations régulières pour répondre au mieux aux contraintes réseaux et capacitaires de l'Éducation Nationale. L'architecture est susceptible d'évoluer.

1. Souveraineté Numérique & Outils Institutionnels

Le développement de la v2 a marqué un tournant architectural majeur : la suppression totale des services cloud commerciaux tiers (Firebase, Google Cloud).

Mentoria s'appuie aujourd'hui exclusivement sur l'infrastructure de l'État :

  • Le Cerveau IA : Albert, développé par Etalab / DINUM (albert.api.etalab.gouv.fr), utilisant des modèles Open Weights hébergés en France.
  • Le Stockage (Nuage) : Les espaces Nextcloud académiques (nuage.apps.education.fr).
  • Le Stockage (Forge) : Le GitLab de l'Éducation Nationale (forge.apps.education.fr).
  • Les Relais & Proxys : Hébergés sur l'incubateur de l'Éducation Nationale (n8n.incubateur.education.gouv.fr).

2. Conformité RGPD "By Design"

L'architecture de Mentoria a été pensée dès sa conception pour garantir une conformité stricte au Règlement Général sur la Protection des Données (RGPD) et assurer la sécurité des informations des élèves. Les choix techniques ont fait l'objet d'échanges et de sollicitations auprès des correspondants des services ministériels (notamment les équipes d'apps.education.fr et de l'incubateur EN) afin de s'assurer que l'outil s'intègre parfaitement dans le cadre de confiance de l'institution.

Aucun traitement de données personnelles côté serveur central

  • Principe de minimisation (Sans compte) : L'élève n'a pas de compte à créer. L'authentification se fait de manière purement déclarative par l'élève (via son identifiant ENT de type prenom.nom) à l'ouverture du lien, minimisant ainsi la collecte de données au strict nécessaire.
  • Absence de Base de Données Tiers : Il n'y a aucune base de données commerciale (SQL/NoSQL) centralisant les bots, les requêtes ou les élèves. Tout est géré sous forme de fichiers plats hébergés au sein du cloud académique souverain de l'enseignant.
  • Durée de vie et purge des logs : Les interactions entre l'élève et l'IA (le "Suivi de classe") sont stockées de façon éphémère dans l'espace Nextcloud personnel de l'enseignant. Elles ont une finalité strictement pédagogique. L'interface offre un outil de purge permettant à l'enseignant de supprimer définitivement ces logs à la fin de la séance, respectant ainsi le droit à l'effacement.

4. Le Relais Souverain (Proxy n8n)

Pour des raisons de sécurité, les navigateurs web appliquent des politiques CORS (Cross-Origin Resource Sharing). Les serveurs de l'API Albert et de Nuage apps.education.fr n'autorisent pas nativement les requêtes directes depuis une page web externe.

La solution architecturale, validée suite aux sollicitations des correspondants techniques ministériels, repose sur des flux n8n hébergés de manière sécurisée sur les serveurs de l'incubateur de l'Éducation Nationale :

Webhook 1 : mentoria_chat (Proxy Albert)

  • Rôle : Recevoir la requête de l'élève (Modèle, Température, Historique), injecter les headers CORS manquants, et relayer la requête vers https://albert.api.etalab.gouv.fr/v1/chat/completions avec le Token Bearer de l'enseignant.

Webhook 2 : mentoria_storage (Proxy Nuage & Logs)

  • Rôle : Transformer les requêtes JSON simples de l'application en requêtes WebDAV complexes pour interagir avec Nextcloud.
  • Fonctions : list, read, write, delete, test.
  • Suivi de classe : Gère l'écriture séquentielle des logs d'interaction de la classe, avec gestion des verrous pour éviter la corruption de fichiers lors d'écritures simultanées de multiples élèves.

5. Architecture du Lien Élève (URL Auto-portée Base64)

Comment l'élève accède-t-il au bot si Mentoria n'a pas de base de données de publication ?

Mentoria utilise un mécanisme d'encodage de la configuration directement dans le hash de l'URL. L'interface compagnion.html est une coquille vide qui se structure dynamiquement à la lecture de son URL.

Flux de publication

  1. L'enseignant clique sur "Publier" dans le Studio.
  2. Le script JS rassemble le titre, le prompt, le corpus textuel, l'avatar, et la clé API chiffrée dans un objet JSON.
  3. Cet objet est compressé en LZ-String puis encodé en Base64.
  4. L'URL finale est générée : https://.../compagnion.html#config=LZStringBase64...

Gestion de l'Avatar (Optimisation de poids)

Puisque l'avatar est encodé en Data-URI (Base64) dans le lien, une image trop lourde produirait une URL qui dépasserait les limites acceptées par les navigateurs ou les ENT (souvent bloqués à 2000-8000 caractères).

Solution : À l'upload, l'image est passée dans un élément <canvas> HTML5, redimensionnée proportionnellement à un maximum de 128x128 pixels, et compressée en JPEG à 82%, réduisant généralement son poids de 500 Ko à ~5 Ko.

6. Traitement IA & Engineering des Prompts

Modèles supportés

Mentoria cible par défaut le modèle openweight-medium de l'API Albert (qui pointe actuellement vers Mistral-Small-3.2-24B-Instruct). Ce modèle offre le meilleur ratio entre vitesse de réponse et fidélité au prompt (indispensable pour l'application stricte des règles de maïeutique).

Extraction du Corpus

L'intégration de fichiers PDF se fait purement côté client via la librairie PDF.js. Seul le texte brut est extrait et concaténé. Une limite stricte de 200 Ko de texte pur est recommandée pour ne pas saturer le lien Base64 ni la fenêtre de contexte du modèle (Context Window).

Limites du corpus

PDF.js extrait uniquement le texte brut (pas les images). 50 pages PDF ≈ 75–100 Ko de texte extrait. Préférer les fichiers TXT exportés depuis le cours.

Niveaux de difficulté (injection dans le prompt)

Le sélecteur Débutant / Standard / Avancé du Studio ajoute une ligne conditionnelle en début de prompt système côté Compagnon :

  • Débutant : NIVEAU DIFFICULTÉ : Débutant — utilise un vocabulaire très simple, beaucoup d'exemples concrets du quotidien, progression pas à pas. Sois très encourageant et patient.
  • Standard : aucune ligne ajoutée (comportement par défaut).
  • Avancé : NIVEAU DIFFICULTÉ : Avancé — utilise un vocabulaire précis et technique, exige de la rigueur dans les réponses, pousse l'élève à justifier et nuancer ses propos.

La valeur est stockée dans le champ config.difficulty de l'objet bot JSON et encodée dans l'URL élève.

Templates de posture

Les 5 templates (Le Socratique, Le Coach, etc.) sont des objets JS côté Studio qui pré-remplissent :

  • Le champ bot-instructions (textarea).
  • Les règles pédagogiques (cases à cocher .rule-toggle).
  • Le niveau de difficulté via setDifficulty().

Ils ne modifient pas l'avatar, le corpus, les boutons rapides ni la clé API. Ils sont purement côté Studio (pas encodés différemment dans l'URL — c'est le résultat de leur application qui est encodé).

Export / Import unitaire (format JSON)

Le fichier .json exporté est exactement l'objet bot stocké en mémoire, sans transformation :

{
  "id": "bot_pink_floyd",
  "updatedAt": "2026-04-23T...",
  "config": {
    "title": "Pink Floyd",
    "instructions": "...",
    "difficulty": "standard",
    "answerSource": "both",
    "pedagogicRules": [...],
    ...
  },
  "corpus": "...",
  "apiKey": "sk-...",
  "vignette": "data:image/jpeg;base64,...",
  "loggingConfig": { "username": "...", ... }
}

À l'import, le fichier est parsé, son id vérifié, puis écrit via writeBotFile() dans le stockage actif (Nuage, Forge ou local). Le manifest est mis à jour en conséquence.

Les Boutons de "Réponse Rapide" (One-Shot Prompting)

Les boutons proposés à l'élève dans le Compagnon utilisent une technique d'injection asynchrone :

  • L'élève clique sur le bouton "Donne-moi un indice".
  • Dans l'interface, le message affiché est "Donne-moi un indice".
  • Mais techniquement, Mentoria envoie le payload suivant à l'IA :
    [DEMANDE PONCTUELLE — exécute UNIQUEMENT cette action, puis reprends ton rôle d'assistant habituel pour la suite] Donne-moi un indice pour avancer sans me donner la réponse directement.

Cette technique force l'IA à agir de manière directive sur un seul échange, sans altérer durablement son "Persona" pédagogique principal.
