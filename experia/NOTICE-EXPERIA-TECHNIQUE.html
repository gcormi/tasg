Notice technique — v0.9 beta

Documentation destinée aux administrateurs, déployeurs et développeurs souhaitant comprendre, installer ou faire évoluer Experia.

Architecture générale

Experia est une application 100 % front-end (HTML/CSS/JS vanilla) sans serveur propre. Elle repose sur trois services externes souverains :

Forge des Communs Numériques Éducatifs (GitLab Pages)

Héberge les fichiers statiques (HTML/CSS/JS). Déploiement automatique via CI/CD à chaque push sur main.

Nuage apps.education.fr (Nextcloud / WebDAV)

Stockage des sessions, résultats et manifest. Chaque professeur utilise son propre compte. Dossier cible : /experia/ dans le Nuage personnel.

Albert API — Étalab (IA générative)

Génère les questions et corrige les réponses ouvertes. Accessible via un proxy n8n hébergé sur l'incubateur de l'Éducation Nationale (contournement CORS).

Flux de données

Professeur → prof.html → n8n proxy Albert → génère activités → Nuage WebDAV (session JSON)

Professeur ← lien partagé (eleve.html#config=BASE64)

Élève → eleve.html → décode config → Nuage (lit session) → joue → Nuage (écrit résultats)

Professeur → tableau résultats → Nuage (lit tous les result_*.json)

Workflow n8n — Experia Storage

Le workflow experia_storage.json fait office de proxy WebDAV. Il accepte des requêtes POST depuis le navigateur (CORS ouvert) et les traduit en requêtes WebDAV authentifiées vers Nuage.

Nœuds

① Webhook

Méthode POST, chemin experia_storage, CORS *, mode réponse : responseNode

↓

② Code Experia

Lit action, username, password, server, filename, content depuis le body. Dispatche selon l'action (test / list / read / write / delete). Utilise this.helpers.httpRequest() pour toutes les requêtes WebDAV.

↓

③ Répondre

Répond avec JSON.stringify($json), Content-Type application/json, headers CORS complets.

Import du workflow

  1. Connectez-vous à votre instance n8n (incubateur Éducation Nationale)
  2. Menu Workflows → Import from File
  3. Sélectionnez experia_storage.json
  4. Activez le workflow (bouton toggle)
  5. L'URL du webhook sera de la forme :
    https://<instance-n8n>/webhook/experia_storage

Actions disponibles (API interne)

POST /webhook/experia_storage
Content-Type: application/json

{
  "action":   "test" | "list" | "read" | "write" | "delete",
  "username": "prenom.nom",
  "password": "mot-de-passe-application",
  "server":   "nuage03.apps.education.fr",  // optionnel, valeur par défaut
  "filename": "session_abc123.json",         // requis pour read/write/delete
  "content":  { ... }                        // requis pour write
}

Logique WebDAV

  • URL de base : https://<server>/remote.php/dav/files/<username>/experia/
  • Authentification : Basic Auth (username + mot de passe d'application)
  • Si PUT retourne 404/405/409 → MKCOL (créer le dossier) puis retry PUT
  • Chaque write/delete met à jour manifest.json (liste de noms de fichiers)

Intégration Albert API

Proxy n8n Albert (Mentoria)

Experia réutilise le proxy Albert développé pour Mentoria. Ce proxy expose un endpoint HTTP qui transmet les requêtes à https://albert.api.etalab.gouv.fr/v1/chat/completions avec la clé API Albert.

Paramètres utilisés

{
  "model": "albert-small",          // ou albert-large selon config
  "messages": [
    { "role": "system", "content": "Tu es un assistant pédagogique..." },
    { "role": "user",   "content": "<prompt généré dynamiquement>" }
  ],
  "temperature": 0.7,
  "max_tokens": 2000
}

Déploiement — Forge GitLab

Pipeline CI/CD (.gitlab-ci.yml)

pages:
  stage: deploy
  script:
    - mkdir -p public
    - cp -r * public/
  artifacts:
    paths:
      - public
  only:
    - main

Tout push sur la branche main déclenche automatiquement le déploiement. Les fichiers sont servis depuis public/ vers l'URL GitLab Pages du projet.

Procédure de mise à jour

  1. Modifier les fichiers HTML localement
  2. git add . && git commit -m "Description" && git push
  3. Attendre 1-2 min (CI/CD) → l'application est mise à jour en ligne
  4. Vider le cache navigateur si nécessaire (Ctrl+Maj+R)

URL de production

L'URL est de la forme : https://<group>.pages.forge.apps.education.fr/<project>/

Consultez les paramètres du projet GitLab → Pages pour l'URL exacte.

Sécurité & RGPD

Transmission des credentials

Point d'attention : Les identifiants Nuage (login + mot de passe d'application) transitent dans le hash URL du lien élève sous forme Base64. Base64 n'est pas un chiffrement — le lien doit être traité comme confidentiel.

Mitigation : Le mot de passe d'application Nextcloud a une portée limitée et peut être révoqué à tout moment depuis Paramètres → Sécurité du Nuage. Toujours utiliser un mot de passe d'application, jamais le mot de passe principal.

Données élèves

  • Aucune donnée nominative collectée — les élèves reçoivent un identifiant aléatoire (Élève-XXXX)
  • Les résultats sont stockés uniquement dans le Nuage du professeur (hébergement EN)
  • Aucune donnée ne quitte le périmètre de l'Éducation Nationale (Nuage + Albert + Forge)
  • Pas de cookies, pas de tracking, pas d'analytics tiers

Clé API Albert

  • La clé API Albert est configurée par le professeur et stockée dans la session URL
  • Elle transite par le proxy n8n hébergé sur l'incubateur EN — elle ne passe pas par un service tiers
  • À terme : la clé devrait être stockée côté serveur dans n8n et ne pas être exposée côté client

Problèmes connus & solutions

Résolu

Erreur écriture code 405 — serveur Nuage incorrect

Symptôme : "Erreur écriture (code 405)" lors de la sauvegarde de session.

Cause : Le serveur par défaut était nuage.apps.education.fr au lieu de nuage03.apps.education.fr.

Fix : URL WebDAV corrigée vers https://nuage03.apps.education.fr/remote.php/dav/files/<username>/experia/. La valeur par défaut dans prof.html est désormais nuage03.apps.education.fr.

Si le problème persiste : Vérifier l'URL WebDAV exacte dans Nuage → Paramètres → Accès WebDAV.

Résolu

Failed to fetch — CORS depuis localhost

Symptôme : "Impossible de joindre le serveur n8n: Failed to fetch" en ouvrant les fichiers HTML localement.

Cause : Le navigateur bloque les requêtes cross-origin depuis file:// ou localhost.

Fix : Tester et utiliser l'application uniquement depuis son URL GitLab Pages (HTTPS).

En cours

Validation end-to-end à confirmer

Le parcours complet (création session → lien élève → jeu → résultats prof) doit être validé en conditions réelles depuis la Forge. Le fix du serveur Nuage a été appliqué mais pas encore testé en production.

Info

Mot de passe d'application vs mot de passe principal

L'authentification WebDAV doit utiliser un mot de passe d'application généré dans Nuage → Paramètres → Sécurité, et non le mot de passe de connexion principal. Le mot de passe principal est rejeté par l'API WebDAV si l'authentification à deux facteurs est activée.
