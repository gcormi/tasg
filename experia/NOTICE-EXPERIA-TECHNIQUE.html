Experia

Notice technique — v1.0

Documentation destinée aux administrateurs, déployeurs et développeurs souhaitant comprendre, installer ou faire évoluer Experia.

Architecture générale

Experia est une application 100 % front-end (HTML/CSS/JS vanilla) sans serveur propre. Elle repose sur trois services externes souverains :

Forge des Communs Numériques Éducatifs (GitLab Pages)

Héberge les fichiers statiques. Déploiement automatique via CI/CD à chaque push sur main.

Nuage apps.education.fr (Nextcloud / WebDAV)

Stockage des sessions et résultats. Chaque professeur utilise son propre compte. Dossier cible : /experia/ dans le Nuage personnel.

Albert API — Étalab (IA générative souveraine)

Génère les questions et corrige les réponses ouvertes. Deux proxys n8n distincts : experia_storage (WebDAV) et experia_albert (IA). Contournement CORS navigateur.

Flux de données

Professeur → prof.html → n8n experia_albert → Albert API → questions JSON → n8n experia_storage → Nuage (session JSON)

Professeur ← lien partagé (eleve.html#config=BASE64)

Élève → eleve.html → décode config → n8n experia_storage → Nuage (lit session) → joue → n8n experia_storage → Nuage (écrit résultats partiels + final)

Professeur → tableau résultats → n8n experia_storage → Nuage (lit result_*.json)

Workflows n8n

Deux workflows distincts sont nécessaires. Les deux ont la même structure : Webhook → Code → Répondre.

1. experia_storage — Proxy WebDAV

Traduit les requêtes POST navigateur en requêtes WebDAV authentifiées vers Nuage (contournement CORS).

① Webhook

POST, chemin experia_storage, CORS *

↓

② Code Experia

Dispatche selon action : test / list / read / write / delete. Basic Auth WebDAV avec username + password. URL base : https://<server>/remote.php/dav/files/<username>/experia/

↓

③ Répondre

JSON.stringify($json), Content-Type application/json, headers CORS complets

// Corps de requête attendu
{
  "action":   "test" | "list" | "read" | "write" | "delete",
  "username": "gcormi",
  "password": "mot-de-passe-application",
  "server":   "nuage03.apps.education.fr",
  "filename": "session_abc.json",    // requis pour read/write/delete
  "content":  { ... }                // requis pour write
}

2. experia_albert — Proxy Albert IA

Transmet les requêtes à l'API Albert avec la clé API fournie dans le corps de la requête.

① Webhook

POST, chemin experia_albert, CORS *

↓

② Code Albert

Lit { action, messages, model, apiKey, temperature } depuis le body. Si action === 'listModels' : GET sur /v1/models. Sinon : POST sur /v1/chat/completions. Header : Authorization: Bearer ${apiKey}

↓

③ Répondre

Même structure que experia_storage

// Corps de requête — génération de contenu
{
  "model": "openai/gpt-oss-120b",
  "messages": [
    { "role": "system", "content": "Tu es un générateur pédagogique..." },
    { "role": "user",   "content": "Génère 10 QCM sur ce corpus..." }
  ],
  "temperature": 0.3,
  "apiKey": "sk-..."
}

// Corps de requête — liste des modèles
{ "action": "listModels", "apiKey": "sk-..." }

Import des workflows

  1. Connectez-vous à votre instance n8n
  2. Menu Workflows → Import from File
  3. Importez experia_storage.json puis experia_albert.json
  4. Activez les deux workflows
  5. Les URLs seront : https://<instance>/webhook/experia_storage et .../webhook/experia_albert

Intégration Albert API

Modèle utilisé

Le modèle par défaut est openai/gpt-oss-120b (Albert Grand). C'est le nom réel de l'API Étalab. Les noms comme albert-small ou albert-large ne sont pas valides.

La liste des modèles disponibles peut être chargée dynamiquement depuis le formulaire de création via le bouton "Charger modèles".

Paramètres de génération

{
  "model": "openai/gpt-oss-120b",
  "messages": [...],
  "temperature": 0.3,    // faible pour des questions précises et factuelles
  "max_tokens": 3000
}

Les 8 types d'activités générées

Type Format JSON attendu Correction
qcm[{q, choices:[4], correct:index, expl}]Automatique (index)
paires[{term, def}]Automatique + pénalité erreurs (−0,5/erreur)
motsmeles[{word}]Automatique (tracé grille)
motscroises[{word, def}]Automatique (complétion valide)
rapidite[{q, choices:[4], correct:index}]Automatique + bonus temps restant
completer[{phrase:"… _____ …", answer}]Levenshtein ≤ 20% de tolérance
intrus[{statements:[4], intrus:index, expl}]Automatique (index)
questions_ia[{q}]Albert (score 0–10 + feedback)

Barème proportionnel

// Score élève converti sur le barème du prof
const profMax = session.scoring[activity];   // barème défini par le prof
const profScore = Math.round((rawScore / rawMax) * profMax * 10) / 10;

Parsing de la réponse Albert

// Albert peut renvoyer du JSON enveloppé dans des balises markdown
const cleaned = content.replace(/```json/gi, '').replace(/```/g, '').trim();
const match = cleaned.match(/\[[\s\S]*\]/);   // extraire le tableau JSON
if (!match) throw new Error('JSON invalide');

Sauvegarde progressive & reprise

Auto-save après chaque activité

autoSaveProgress() est appelée à chaque fois qu'un élève termine une activité (saveScore()). Elle écrit silencieusement un fichier résultat partiel sur le Nuage avec partial: true. Si la sauvegarde échoue, l'erreur est ignorée (l'élève n'est pas bloqué).

Reprise de session

À la connexion, doConnect() cherche un fichier résultat existant pour cet élève sur cette session. Si trouvé, les scores sont rechargés dans E.scores avant d'afficher le dashboard. L'élève retrouve ses activités déjà complétées marquées "terminé" et peut continuer les restantes.

// Reprise — lecture du résultat partiel existant
const filename = `result_${session.id}_${studentId.replace(/[^a-zA-Z0-9]/g,'-')}.json`;
const prev = await storage({ action: 'read', filename });
if (prev.success && prev.content?.scores) {
  E.scores = prev.content.scores;   // restaurer les scores
  E._resuming = true;
}

Condition de reprise : l'élève doit saisir exactement le même identifiant (sensible à la casse) que lors de sa première connexion.

Import de fichier HTML comme corpus

Lorsqu'un fichier .html ou .htm est importé, le texte est extrait via un élément DOM temporaire :

const tmp = document.createElement('div');
tmp.innerHTML = htmlContent;
// Supprimer les éléments non textuels
tmp.querySelectorAll('script, style, noscript, svg, img, head').forEach(el => el.remove());
// Extraire le texte bloc par bloc
const text = Array.from(tmp.querySelectorAll('p, li, h1, h2, h3, h4, h5, h6, td, th, div, br'))
  .map(el => el.textContent.trim())
  .filter(t => t.length > 0)
  .join('\n');

Le texte résultant ne contient aucune balise HTML, attribut, CSS ou JavaScript. Seul le contenu textuel visible est conservé, avec des sauts de ligne entre les blocs.

Persistance locale (localStorage)

Seuls trois champs sont mémorisés dans localStorage sous la clé experia_config :

{ "user": "gcormi", "pass": "mot-de-passe-app", "apiKey": "sk-...", "server": "nuage03..." }

La déconnexion depuis l'interface vide explicitement ces champs (sécurité pour ordinateurs partagés).

Sécurité & RGPD

Credentials dans le lien élève

Point d'attention : Les identifiants Nuage (login + mot de passe d'application), la clé Albert et les URLs n8n sont encodés en Base64 dans le hash URL du lien élève. Base64 n'est pas un chiffrement.

Mitigation : Utiliser un mot de passe d'application (portée limitée, révocable) et non le mot de passe principal. Le lien doit être traité comme confidentiel.

Données élèves

  • Les élèves s'identifient avec leur identifiant ENT — aucune donnée supplémentaire collectée
  • Les résultats sont stockés uniquement dans le Nuage du professeur (hébergement EN)
  • Aucune donnée ne quitte le périmètre de l'Éducation Nationale
  • Pas de cookies, pas de tracking, pas d'analytics tiers

Suppression des données

La suppression d'une session depuis prof.html efface le fichier session et tous les fichiers résultats élèves associés sur le Nuage. Aucune donnée résiduelle.

Problèmes connus & solutions

Info

Mot de passe d'application obligatoire

L'authentification WebDAV doit utiliser un mot de passe d'application généré dans Nuage → Paramètres → Sécurité. Le mot de passe principal ENT est rejeté si l'authentification à deux facteurs est activée.

Info

Serveur Nuage variable selon l'académie

Le serveur Nuage par défaut est nuage03.apps.education.fr mais certains profs sont sur nuage01, nuage02, etc. Vérifiable dans Nuage → Fichiers → Paramètres WebDAV. Modifiable via "Paramètres avancés" dans l'écran de connexion.

Attention

Failed to fetch — ouverture locale

Symptôme : "Impossible de joindre le serveur n8n: Failed to fetch" en ouvrant les HTML localement.

Cause : Navigateur bloque les requêtes cross-origin depuis file://.

Fix : Toujours utiliser l'URL GitLab Pages (HTTPS).

Info

Reprise de session — identifiant sensible à la casse

Le nom du fichier résultat est construit à partir de l'identifiant élève. Si l'élève saisit Marie.Dupont la première fois et marie.dupont la suivante, le système considère un nouvel élève. Recommander une convention uniforme dans la classe.
